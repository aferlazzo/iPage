About

The spriteMan web page was written by Anthony Ferlazzo out of frustration with the lack of graphic web development tools.

In a perfect world one would come up with an idea for a web page, write the HTML code, the CSS, the javascript and server pieces, and then move on to the next assignment. But one then has to also assume that there are no specification mistakes, changes of direction, or anything else that could possibly go wrong. The real world of web development is not quite so tidy a place. For us, that means there are changes introduced by internal and external factors, if not during development then over time.

This web page helps in developing, maintaining, and improving HTML pages that use sprites. Notice the active-verb choices in the previous sentence. That's because there is rarely a 'development' or an 'improvement' event. Rather we are often developing and improving via a process called 'continuous improvement'. Frequently referred to as 'tweaking' because we begin with a vision of a HTML page, but one we achieve that vision we realize that it is wanting for a change here and an adjustment there. In fact as this page was being developed, how it was going to look and what it was going to do was continually being refined.

Along with the HTML code, the graphic objects and other formating decisions like paragraph placement and so forth are being fine-tuned. That's why the benefit of creating sprites during the initial web page development for page load speed and memory savings have the unwanted side-effect of making page 'tweaking' a more complex endeavor.

Creating a sprite at any point in the HTML page life-cycle results in efficiencies, but it also creates yet another thing that must be adjusted when a web page later changes. unfortunately there are few tools that help the person who is charged with making such a change to rebuild or adjust sprites, save for completely recreating the sprite from individual graphic files.

In fact spriteMan is a misnomer. Rather than managing sprites, what it actual does is create the CSS code for a particular item within a sprite. For anyone who has received a sprite and is then asked to make adjustments to it or to the way it is positioned on a page, spriteMan will be useful.

Should it be called spriteDecoder, or something else instead?

Help

Enter the pathname to a local or remote sprite image file on the 'Sprite File' tab. It will appear within a grid. Next to it will appear a copy of the sprite without a grid. You can drag this copy around the page if needed for a better view of the selection.

Choose a selection background to colorize the selected portion of the sprite. Initially the entire sprite will be selected. You may use the grid frame's coordinates as handles for adjusting the selection rectangle. Likewise, you can directly adjust the selection by dragging the rectangle or by resizing it.

The 'Coordinates' tab is yet another place to adjust the sprite grid and the selected item on it.

Some sprites may be too large to view along with the selected item to the right. Click on one of the corners to reduce the sprite's size. A red corner means the height or width of the sprite is partially hidden. Clicking again will restore the sprite's height or width to full view.

The 'Item CSS' tab contains the css code to use once you have selected an item. This is an accurate representation of the item and is the result to use on your web pages.
